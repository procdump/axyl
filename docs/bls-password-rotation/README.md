# Rotating a Node's BLS Keystore Passphrase

Change the passphrase that encrypts a node's BLS private key (`bls.kw`) **without
changing the node's identity**. Use this when a passphrase is suspected compromised, when
rotating secrets on a schedule, or to migrate a keystore that was encrypted with a
**weak (1-round) PBKDF2** count to the standard **1,000,000-round** scheme —
re-encrypting upgrades the work factor automatically.

The node's **BLS identity does not change**: same public key, same derived libp2p network
keys, same peer ID, same stake. **No re-staking or on-chain action is required.**

---

## Background — what is and isn't being rotated

When a node is provisioned, `keytool generate` creates three files under
`<datadir>/node-keys/`:

| File           | Contents                                                        |
| -------------- | --------------------------------------------------------------- |
| `bls.kw`       | The BLS **private key**, encrypted with the passphrase (Base58) |
| `primary.seed` | Seed string for deriving the primary libp2p network keypair     |
| `worker.seed`  | Seed string for deriving the worker libp2p network keypair      |

The encryption scheme (`crates/infrastructure/config/src/keys.rs`):

- **KDF:** PBKDF2-HMAC-SHA256, 12-byte random salt. The iteration count is the
  brute-force work factor:
  - **1,000,000 rounds** — the production strength; always used when writing.
  - **1 round** — effectively *no* protection; a weak passphrase is crackable in
    seconds. Keys generated by builds that accidentally enabled the `test-utils`
    feature are 1-round. The binary still **reads** these (legacy fallback), so
    rotation transparently upgrades them to 1,000,000 rounds.
- **Cipher:** AES-256-GCM-SIV, 12-byte random nonce
- **On-disk layout of `bls.kw`** (after Base58 decode):
  `salt[0..12] || nonce[12..24] || ciphertext[24..]`

**Rotation only re-encrypts `bls.kw`.** The BLS private key bytes are unchanged,
therefore:

- The **BLS public key** (validator identity) is unchanged.
- The **primary/worker network (libp2p) keypairs** are unchanged — they are derived
  deterministically from the *private key* + seed strings, not from the passphrase.
  **Do not touch `primary.seed` / `worker.seed`.**
- **No on-chain action** is required: stake, proof-of-possession, and peer ID all stay
  valid.

> ⚠️ **Never use `keytool generate` to "reset the password."** It mints a *brand new*
> BLS keypair, which changes the validator identity and would require re-staking and
> network re-registration. `generate` requires `--force` to overwrite existing keys —
> that flag is your last warning.

> ⚠️ **Rollout order matters.** Compatibility is one-directional: a new binary reads old
> (1-round) keystores, but an old binary **cannot** read a keystore rewrapped at
> 1,000,000 rounds. **Upgrade the node image/binary first, rotate second.**

## The `rotate-passphrase` command

Rotation is done with `keytool rotate-passphrase`
(`crates/infrastructure/network-cli/src/keytool/rotate_passphrase.rs`). It:

- takes the **current** passphrase from `--bls-passphrase-source` (the same source you
  use to start a node: `env` → `RL_BLS_PASSPHRASE` (default), `stdin`, or `ask`);
- takes the **new** passphrase from the `RL_BLS_NEW_PASSPHRASE` environment variable if
  set, otherwise prompts for it interactively (with confirmation); it must be non-empty
  — **choose a strong, random passphrase**;
- re-encrypts the same private key with a fresh salt + nonce at 1,000,000 rounds and
  atomically replaces `bls.kw` (write-temp + rename — never leaves a torn file);
- re-reads the keystore with the new passphrase to confirm it decrypts before
  reporting success.

A `--dry-run` flag only verifies the current passphrase decrypts the keystore and logs
the BLS public key, without writing anything — use it as the pre-check.

## Building the binary

From the repository root:

```bash
cargo build -p rayls-network --bin rayls-network --release
# binary at target/release/rayls-network
```

> ⚠️ **Always include `-p rayls-network`.** Building with just
> `cargo build --bin rayls-network` (no `-p`) at the workspace root resolves features
> across the whole workspace and **leaks the `test-utils` feature into the binary**,
> silently downgrading PBKDF2 to **1 round** when writing keystores. The `-p` flag
> scopes feature resolution to the binary's own dependency graph. This is the same flag
> the production `etc/docker-network/Dockerfile` uses.

Or build via Docker for prod parity:

```bash
docker build -f etc/docker-network/Dockerfile -t rayls-node .
# the binary inside the image is /usr/local/bin/rayls
```

Prerequisites for a host build: a recent stable Rust toolchain (prod image uses 1.91;
<https://rustup.rs>) and, on Debian/Ubuntu,
`build-essential cmake libclang-dev pkg-config libssl-dev libapr1-dev`.

## Procedure

The keystore lives at `<datadir>/node-keys/bls.kw`; `--datadir` must point at the
**parent** of `node-keys` (e.g. for `/var/lib/rayls/node-keys/bls.kw`, use
`--datadir /var/lib/rayls`).

### Pre-checks

1. Confirm the **current** passphrase decrypts the keystore (read-only):

   ```bash
   RL_BLS_PASSPHRASE='<current-passphrase>' rayls-network \
     keytool rotate-passphrase --dry-run --datadir <datadir>
   ```

   On success it logs `dry-run: passphrase decrypts the BLS keystore` with the
   `bls_public_key` — record that value; it must be identical afterward. If it errors
   with `Could not decrypt BLS key`, **stop** — rotation is impossible without the
   correct current passphrase (you'd need to restore from a key backup).

2. Note where the node sources its passphrase at runtime (`--bls-passphrase-source`:
   `env` → `RL_BLS_PASSPHRASE`, `stdin`, or `ask`) — you'll update that in step 4.

### Step 1 — Stop the node

Rotation rewrites `bls.kw`; take the node offline first.

```bash
# example — adapt to your orchestration (docker compose / systemd / etc/manage scripts)
docker compose stop <node-service>
```

### Step 2 — Back up the current keystore

```bash
cp -a <datadir>/node-keys/bls.kw <datadir>/node-keys/bls.kw.bak.$(date +%Y%m%d%H%M%S)
```

Keep this backup until rotation is verified. It is your rollback (still readable with
the **old** passphrase).

### Step 3 — Rotate

Scripted (both passphrases from the environment):

```bash
RL_BLS_PASSPHRASE='<current-passphrase>' \
RL_BLS_NEW_PASSPHRASE='<new-strong-passphrase>' \
rayls-network keytool rotate-passphrase --datadir <datadir>
```

Interactive (prompts for both, new passphrase confirmed twice):

```bash
rayls-network --bls-passphrase-source ask \
  keytool rotate-passphrase --datadir <datadir>
```

Success logs `BLS keystore re-encrypted with the new passphrase` with the
`bls_public_key` — compare it against the pre-check value.

### Step 4 — Update the secret store

Update wherever the runtime passphrase comes from so the node starts with the **new**
value:

- **env:** update `RL_BLS_PASSPHRASE` in the vault / secrets manager / `.env` /
  k8s Secret that feeds the container. In docker compose list-form `environment:`
  entries, do **not** quote the value — quotes become part of the passphrase
  (`- RL_BLS_PASSPHRASE=NewPass`, not `- RL_BLS_PASSPHRASE="NewPass"`).
- **stdin:** update the file/secret that gets piped in.
- **ask:** brief the operator who enters it interactively.

### Step 5 — Restart and verify

```bash
docker compose start <node-service>
```

Verify:

1. The node decrypts `bls.kw` and starts (no `Could not decrypt BLS key` error).
2. The BLS public key in the logs matches the pre-check value.
3. The node connects to peers and participates in consensus normally (peer ID is
   unchanged, so reconnection should be seamless).

### Step 6 — Clean up

Once the node is confirmed healthy on the new passphrase:

```bash
shred -u <datadir>/node-keys/bls.kw.bak.*   # or secure-delete equivalent
```

Also purge the **old** passphrase from any secret store history if your tooling retains
versions.

---

## Rollback

If the node fails to start after rotation:

```bash
docker compose stop <node-service>
cp -a <datadir>/node-keys/bls.kw.bak.<timestamp> <datadir>/node-keys/bls.kw
# restore the OLD passphrase in the secret store
docker compose start <node-service>
```

The backup is the original encrypted file, so restoring it + the old passphrase returns
the node to its exact prior state.

## Quick reference

| Item                          | Value                                                                            |
| ----------------------------- | -------------------------------------------------------------------------------- |
| Build command                 | `cargo build -p rayls-network --bin rayls-network --release`                     |
| Keystore file                 | `<datadir>/node-keys/bls.kw` (Base58)                                            |
| KDF                           | PBKDF2-HMAC-SHA256, 12-byte salt; writes 1,000,000 rounds, reads legacy 1-round  |

| Files NOT changed by rotation | `primary.seed`, `worker.seed`                                                    |
| On-chain impact               | None (identity preserved, no re-staking)                                         |
| Code: encrypt/decrypt         | `crates/infrastructure/config/src/keys.rs`                                       |
| Code: keytool subcommand      | `crates/infrastructure/network-cli/src/keytool/rotate_passphrase.rs`             |
